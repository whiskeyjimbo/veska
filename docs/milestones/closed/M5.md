# M5 - Optional Review Pipeline

**Goal:** An optional LLM-driven review pipeline (security,
contract drift) runs as a goroutine after promotion. Off by default.
Honest cost story in the docs. Findings surface via MCP; the
human-action gate applies.

**Dependencies:** M3.

## Exit gates

- [x] Review pipeline disabled by default. Enabling requires explicit `review.enabled = true`. → `solov2-nz2.2` (config default `review.enabled=false`; no review row enqueued when disabled).
- [x] V2.0 ships **one** `LLMGenerator` impl: `ollama`. Setting `provider` to anything else fails fast at startup. Hosted-provider impls (`anthropic`, `openai`, `gemini`, `openai_compatible`) and the env-var-key resolution they require move to V2.0.1. → `solov2-nz2.1` (startup gate refuses non-`ollama` provider).
- [x] `veska doctor egress` reports the local Ollama endpoint whenever review is enabled. → `solov2-jcq` (egress lists a `review_llm` destination when `review.enabled`).
- [x] Per-promotion token-budget cap enforced; refusal carries `degraded_reasons: ['review_quota_exceeded']`. Tested with a synthetic over-budget promotion. → `solov2-nz2.5`.
- [x] Documented per-commit *time* budget for one review of a 100-file commit on the reference laptop (m5.00.1). → `solov2-zrf` - measured 2026-05-18: **1h35m31s** for 100 files (~57.3s/file, `llama3`/CPU); see Close report.

## Epics

### Epic m5.00 - Local `LLMGenerator` impl (Ollama)

**Tracker:** `solov2-nz2.1` (closed)

**DoD**
- [x] HTTP-based `LLMGenerator` impl in `internal/infrastructure/llm/ollama/`. Posts to a local Ollama endpoint configured via `[llm_generator].endpoint` and `[llm_generator].model` (CONFIG-SURFACE.md).
- [x] Composition root selects this impl unconditionally at V2.0; setting `provider` to anything other than `"ollama"` fails fast at startup with a clear error pointing at the V2.0.1 deferral.
- [x] Each call populates `Generation.Provenance` with `(model_id, prompt_template_version, input_hash)` per SOLO-05 §2.5 so cached outputs can be invalidated when the model changes.
- [x] Per-impl timeout (`llm_generator.timeout` DEFAULT) and 3-retry policy on transient failures (5xx, network).
- [x] Documented time-budget example for one review of a 100-file commit on the reference laptop. Lives in M5 close report.

**Acceptance**
- Test: starting the daemon with `provider = "anthropic"` (or any non-`ollama` value) exits non-zero with a message naming the V2.0.1 deferral.
- Test: a synthetic review against the local Ollama returns a populated `Generation` with a non-empty `Provenance`.
- Test: a transient 5xx from Ollama triggers exactly 3 retries with exponential backoff before the post_promotion_queue row is marked failed.

**Sub-issues**

**Deferred to V2.0.1** (filed as separate beads issues, not blocked by M5 close):

### Epic m5.01 - Review goroutine

**Tracker:** `solov2-nz2.2` + `solov2-nz2.3` (closed)

**DoD**
- [x] post-promotion queue `work_kind='review'`; one goroutine drains it.
- [x] `LLMGenerator` interface dispatches to local Ollama or configured remote endpoint.
- [x] Failure retries up to 3.
- [x] On persistent failure, the row stays `failed` (no auto-ack)

  `rule='review-pipeline-failure'`, `severity='high'`,

## Close report

### Exit gate 5 - per-commit review time budget (m5.00.3 / `solov2-zrf`)

**Measurement procedure.** The repeatable harness lives at
`tools/loadtest/reviewtiming/` and is run via:

```bash
make eval-review-timing
```

It builds a synthetic ~100-file Go commit fixture, drives each file through the
real `internal/application/review` `Handler` (the `WorkKindReview` lane) against
a real local Ollama generator, and reports the total wall-clock budget for the
100-file pass plus the per-file mean. Override `REVIEW_TIMING_FILE_N`,
`VESKA_OLLAMA_URL`, and `VESKA_REVIEW_MODEL` as needed; the harness skips
cleanly when Ollama is unreachable. See `tools/loadtest/reviewtiming/README.md`.

**Reference-laptop time budget (measured 2026-05-18).** One review pass over a
synthetic 100-file commit, `llama3` on local CPU Ollama:

| Metric | Value |
|---|---|
| Model | `llama3` (local Ollama, CPU) |
| Files | 100 |
| Wall-clock budget | **1h35m31s** (`total_ms` 5,731,446) |
| Per-file mean | **~57.3s** (`per_file_mean_ms` 57,314) |
| Reviewed OK | 99 / 100 |
| Failed | 1 (one `contract_drift` response had a finding object missing `message` - parser robustness follow-up `solov2-spb`) |

This is the honest cost story: an LLM review pass over a large commit on local
CPU Ollama is a ~1.5-hour background job. The review pipeline is off by default
(`review.enabled=false`) and runs as a post-promotion goroutine - it never
blocks promotion or the commit hook. Operators who enable it on commodity
hardware should expect review latency on this order; a GPU host or a smaller
per-commit file set scales it down roughly linearly (~57s/file).
