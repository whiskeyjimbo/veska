# M7 — Vuln + Secrets Scanning

**Goal:** Ship the two promotion checks `SOLO-11 §2.1` marks PLANNED —
`vuln-scan` and `secrets-scan`. After M7, a commit that adds a
CVE'd dependency or a secret-shaped string surfaces a `Finding` on the
MCP surface. No network egress on the promotion path.

**Tracker:** epic `solov2-2ps`.

**Dependencies:** M3 (promotion-check pipeline). Full implementation
design: [`M7-design-vuln-secrets.md`](../M7-design-vuln-secrets.md).

## Exit gates

- [x] A `go.mod` with a known-CVE'd dependency produces a `vuln` finding. → `solov2-2ps.5` (vulnscan check test, fake VulnSource).
- [x] A commit adding a secret-shaped string produces a `secret_leak` finding with a redacted snippet. → `solov2-2ps.10`.
- [x] Vuln-scan touches no network at promotion time. → `solov2-2ps.3` (offline-Scan network-block test).
- [x] Secrets-scan flags only added lines, never pre-existing ones. → `solov2-2ps.7` + `.10` (`AddedLines` seam carries added lines only).
- [x] `veska doctor egress` reports the OSV endpoint when `[vuln_source]` is configured. → `solov2-2ps.6`.
- [x] `make all` clean. → build/test/vet/lint/layercheck all green (`make all` exits 0). The pre-existing staticcheck/unused lint baseline was cleared in `solov2-rdw`.

## Epics

### Epic m7.01 — Vuln scanning

**Tracker:** `solov2-2ps.1`–`.6`

**DoD**
- [x] `ports.VulnSource` reshaped to `Refresh()` + `Scan([]Dependency) []VulnFinding`; `NullVulnSource` updated.
- [x] `go.mod` manifest reader returns the full dependency set (direct + indirect) via `golang.org/x/mod/modfile`.
- [x] `osv` adapter: `Refresh()` downloads OSV's full Go-ecosystem advisory dump to `~/.veska/cache/osv/`; `Scan()` matches deps against the local dump with no network.
- [x] Daemon-owned refresh goroutine on a 24h configurable cadence, kicked on start; cold start is silent catch-up.
- [x] `vulnscan` check emits `vuln` findings (`source_layer = security`), anchored on `go.mod` with `key = advisoryID+package`.
- [x] Wired in `wire.go`; `[vuln_source] provider = "osv"` config; `veska doctor egress` reports the OSV endpoint. Ships off by default.

**Sub-issues**
- `solov2-2ps.1` — reshape `VulnSource`; `Dependency`/`VulnFinding` types; update `NullVulnSource`.
- `solov2-2ps.2` — `go.mod` reader → `[]Dependency`.
- `solov2-2ps.3` — OSV.dev adapter with disk cache (depends on .1).
- `solov2-2ps.4` — daemon refresh goroutine (depends on .3).
- `solov2-2ps.5` — `vulnscan` promotion check (depends on .1, .2, .3).
- `solov2-2ps.6` — `wire.go`, config, egress (depends on .3, .5).

### Epic m7.02 — Secrets scanning

**Tracker:** `solov2-2ps.7`–`.11`

**DoD**
- [x] `checks.Input` carries an `AddedLines` seam, populated once by the promotion path.
- [x] `ports.SecretsScanner` (`Scan(ScanInput) []SecretFinding`) plus `ScanInput`/`SecretFinding` types.
- [x] `veska-builtin` impl: regex ruleset + Shannon-entropy heuristic, redaction built in.
- [x] `secretsscan` check emits `secret_leak` findings (`source_layer = security`) with rule + redacted snippet, anchored on the file with `key = rule+line`.
- [x] Wired in `wire.go`, ships on by default; `[promotion]` config gates per-check enable/disable.

**Sub-issues**
- `solov2-2ps.7` — extend `checks.Input` with `AddedLines`; promotion path populates it.
- `solov2-2ps.8` — `SecretsScanner` port + `ScanInput`/`SecretFinding` types.
- `solov2-2ps.9` — `veska-builtin` regex + entropy impl (depends on .8).
- `solov2-2ps.10` — `secretsscan` promotion check (depends on .7, .8, .9).
- `solov2-2ps.11` — `wire.go`, `[promotion]` per-check config (depends on .10).
