# M3 - Pipelines and Embedder

**Goal:** Promotion pipeline runs structural checks synchronously; an
async embedder worker keeps `node_embedding_refs` drained;
`semantic_search` runs against the production vector path (sqlite-vec
default, usearch/float16 above the M2-ratified threshold per
ADR-S0015); auto-link suggestions land;
revalidation sweeps invalidate stale findings on content drift.

**Dependencies:** M1, M2 (incl. m2.06 dual-backend ratification).

## Exit gates

- [x] **Gate 1 - Embedder throughput on local CPU Ollama: 5–20 emb/s.** MET at **8.43 emb/s** on reference box (`nomic-embed-text` via `tools/loadtest/embedder`, commit `f91b51e`; worker rate-limit was the bound - Ollama capacity is higher).
- [x] **Gate 2 - `semantic_search` recall measured at 50k and 250k.** Both MET on the real-Ollama path with `nomic-embed-text` (`tools/loadtest/recall`): **50k - mean_recall=0.99, p95=137.5ms** (sqlite-vec backend, commit `cabf934`); **250k - mean_recall=1.00, p95=148.91ms** (usearch/float16 HNSW backend, solov2-wqi). Both exceed ADR-S0014's M1 substrate bench (0.987@50k / 0.954@250k); M3 confirms the number end-to-end on the live `semantic_search` path.
- [x] **Gate 3 - Auto-link false-positive rate on a fixture.** MET: **FP rate 0.083%** (25 false positives in 30,000 candidates) at similarity threshold 0.50 on a 6,000-node real-Ollama fixture (`VESKA_CORPUS=semantic`, `tools/loadtest/autolink` + `synthcorpus.GenerateSemanticCorpus`). FP rate is 0.000% at thresholds ≥ 0.55. Reaching a measurable number took three fixes, each a real defect surfaced in the process: (a) the original `synthcorpus` text was tuned for `FakeEmbed` near-duplicates - replaced with 30 hand-authored disjoint topic vocabularies that nomic-embed-text actually clusters (within-cluster cosine ≈0.74 vs between ≈0.49); (b) `sqlite-vec` linear scan is O(N²) and timed out at scale - the harness now selects the usearch/float16 HNSW backend via `VESKA_VECTOR_BACKEND=usearch` (build tag `hnsw_native`, libusearch_c.so v2.25.2); (c) **production embedder does not L2-normalize** - nomic vectors have norm ≈19, collapsing autolink's `score=1/(1+L2dist)` to ≈0.005 and emitting zero candidates at any usable threshold. The fixture generator now normalizes; the production fix landed in **`solov2-uug`** (commit `fb6ffcd`: `embedder.Worker` L2-normalizes every vector before storage, and `autolink.DefaultThreshold` was retuned 0.85 → 0.60 against these numbers).
- [x] **Gate 4 - Revalidation completes in <60s for a 10k-edge commit.** MET at 6.4s (commit `15e5e4c`).
- [x] **Gate 5 - All three M3 post-promotion queue `work_kind` lanes (`embed`, `auto_link`, `revalidate`) drain to `state='done'` under a fuzz workload.** MET: `TestQueueFuzz` drains 100 promotions × 3 lanes in ~17s with `budget_met=true`, `stuck_per_kind={0,0,0}` (commit `da9a0f6`). The fourth `WorkKind` enum value (`review`) is reserved for M5 and is intentionally out of scope here; the gate-5 fuzz workload does not enqueue `review` rows.

### Epic m3.04 - Auto-link

**Tracker:** `solov2-64k.4` (closed)

**DoD**
- [x] post-promotion queue `work_kind='auto_link'` proposes `Edge` rows with `confidence='unresolved'` from embedding similarity AND emits one finding per proposed edge (`source_layer='semantic'`) whose anchor is that specific edge. The edge is never inserted silently - it only becomes visible via the finding.
- [x] Threshold configurable; default tuned on the eval set.
- [x] No auto-merge; the developer accepts or suppresses.
- [x] Accept-flow uses existing `eng_close_finding` with
  `reason="accept"`; the close handler promotes the anchor's
  unresolved edge to `confidence='definite'`. Suppress/close paths
  leave the edge at `confidence='unresolved'` (visible only to
  queries that opt into unresolved edges). No new MCP tool.

**Sub-issues**
- `m3.04.1-similarity` - top-k similarity per node.
- `m3.04.2-finding-shape` - auto-link finding kind.
- `m3.04.3-accept-flow` - `eng_close_finding[reason="accept"]` promotes the linked edge.
- `m3.04.4-fp-measurement` - false-positive rate on fixture.

### Epic m3.05 - Revalidation

**Tracker:** `solov2-64k.5` (closed)

**DoD**
- [x] post-promotion queue `work_kind='revalidate'` re-checks open findings whose anchor `content_hash` changed.
- [x] Per-rule dispatch on stale findings:
  - `dead-code`, `contract-drift`: cheap re-run on current node state. If the rule still fires, REFRESH `anchor_content_hash` in place (state stays `open`, no closed_reason). If the rule no longer fires, transition to `closed` with `closed_reason='revalidated_obsolete'`.
  - `auto-link`: always close as `revalidated_obsolete` (similarity re-run is heavy and out of scope for m3.05).
  - any other rule: conservative close as `revalidated_obsolete`.
- [x] No `closed_reason='superseded_by_revalidation'` is ever written. `finding_id` = `hash(rule, anchor)` is branch-stable, so a re-firing rule produces the SAME row - refreshing the anchor hash is the supersede path. (SOLO-04 §8.1.)
- [x] Sweep runs after promotion; bounded to the symbols touched by the promotion.

**Sub-issues**
- `m3.05.1-anchor-tracking` - finding ↔ node hash linkage.
- `m3.05.2-sweep` - revalidate goroutine.
- `m3.05.3-supersede` - chain old → new finding.
- `m3.05.4-bench` - 10k-edge commit revalidation timing.
