# M0 — Substrate Spike

**Goal:** Empirically validate the two substrate decisions M1
will be built on top of — sqlite-vec as the vector index, and
branch-in-PK as the row-key shape — **before** any M1
implementation work begins.

Both decisions rest on measurements that have not been taken.
ADR-S0001 names sqlite-vec as the entire vector substrate;
SOLO-08 §4 puts `branch` in the PK of `nodes`, `edges`, and
`findings`. M0 takes both measurements with throwaway harnesses.
If either misses, the relevant ADR or schema is amended with the
failure data in hand — not under M1 schedule pressure once code
is built on top.

This milestone ships **no production code**. It produces numbers
and decisions. Both spikes use synthetic loaders against the
target schema; neither needs the parser, the promotion pipeline, or
the application layer.

**Dependencies:** none. M0 is the entry gate to M1.

## Reference hardware

8-core, 16 GiB RAM, NVMe SSD. M1 MacBook Air or equivalent x86-64
laptop. Numbers measured on this tier are normative.

## Exit gates

Two spikes, two decision matrices in §Outcomes. Numbers from
both feed `docs/docsv2solo/design/13-nfr/README.md` §3 (the
matching `BUDGET (unmeasured)` rows are relabelled
`BUDGET (measured M0)` with the spike commit hash) and are cited
from the M0 close commit.

### Vector substrate (vec0)

- [ ] **Latency at 50k vectors.** Warm p95, p99, max for k=10
  queries over 50 000 768-dim float32 vectors loaded into a
  `vec0` virtual table. Cold (post-restart) p95 separately.
- [ ] **Latency at 1M vectors.** Same query shape, 1 000 000
  vectors. Warm p95 and cold p95.
- [ ] **Recall at 50k and 1M vectors.** Recall@10 and recall@50
  against a 100-pair hold-out generated from the same vector
  population. The recall@10 ≥ 0.95 floor at 50k is a working
  number; the harness either confirms it or names the actual
  number with the curve.
- [ ] **Resource cost.** Daemon RSS during query and during load;
  on-disk size of the `vec0` shadow tables at 50k and 1M.
- [ ] **Vec0 ceiling.** Node count at which brute-force vec0
  first crosses either the SOLO-13 §3.1 `semantic_search` p95
  budget or the §3.3 soft RSS cap (2 GiB) at steady state.
  Sweep populations between 50k and 2M to find the crossover;
  the ceiling is the smaller of the two crossover counts. This
  is the input SOLO-13 §3.3.1's gate is keyed against.

### Branch-in-PK schema (OQ-S006)

- [ ] **Row growth.** Total `nodes`, `edges`, and `findings` row
  counts at synthetic populations of 50 branches × 100k symbols
  with a configurable per-branch dirty-symbol overlap (10%, 30%,
  60% of base set differs per branch).
- [ ] **Disk size.** Live `engram.db` size at the same populations,
  before and after `engram gc --branches` sweeps deleted branches.
- [ ] **Query p95.** `eng_get_node`-shaped indexed lookup p95
  against the multi-branch population (warm); `eng_get_edges`-shaped
  src+branch+kind index lookup p95.
- [ ] **GC sweep cost.** Wall-clock and disk-reclaim for
  `engram gc --branches` against a 50-branch population with 10
  branches deleted.

## Outcomes

### Vector substrate

The spike must produce **three** numbers per population: warm
p95, recall@10 against a 100-pair hold-out, and the **vec0
ceiling** (the node count at which brute-force vec0 first
crosses either the SOLO-13 §3.1 `semantic_search` budget or the
§3.3 soft RSS cap on the reference laptop). The ceiling is the
input SOLO-13 §3.3.1's gate is keyed against.

| Spike result | Action |
|---|---|
| **Green.** 50k warm p95 ≤ 100ms AND recall@10 ≥ 0.95; 1M recall@10 ≥ 0.85; 1M warm p95 ≤ 200ms; **vec0 ceiling ≥ 250 000 nodes**; RSS / disk reasonable. | ADR-S0001 confirmed. OQ-S001 marked resolved with the measured numbers. OQ-S003 keeps its M3 gate. M1 proceeds. No fallback ADR is written for M1. |
| **Yellow.** 50k passes; 1M is borderline (recall in 0.75–0.85 OR p95 in 200–400ms); ceiling ≥ 250 000 nodes. | ADR-S0001 confirmed for V2.0. A numeric kill criterion is set in OQ-S003 informed by the actual curve, gated at M3. M1 proceeds. The fallback ADR is written ONLY if M3 trips the criterion. |
| **Red — quality.** 50k misses 100ms p95 OR 50k recall < 0.95 OR 1M recall < 0.75 OR resource costs are unworkable. | ADR-S0001 is amended (or retracted in favor of a new ADR). The fallback substrate is chosen against measured failure data — candidates include `lancedb` (embedded HNSW, Go bindings) and `hnswlib` via cgo. The M1 plan is updated before M1 work resumes. |
| **Red — ceiling.** Vec0 ceiling falls below **250 000 embedded nodes** on the reference laptop, regardless of how 50k/1M numbers look. | The HNSW pivot ADR (OQ-S003) is promoted from M3 work to M1 work and is **mandatory before M1's m1.03 begins**. ADR-S0001 is amended to cite the ceiling and reference the pivot ADR. SOLO-13 §3.3.1 is updated with the measured ceiling number. The pivot ADR specifies the chosen backing, the recall floor, and how it preserves the single-tarball backup story. |

### Branch-in-PK schema

| Spike result | Action |
|---|---|
| **Green.** Row growth scales linearly with `branches × dirty-symbols-per-branch` and stays under a defensible disk budget (target: ≤ 5 GiB at 50 branches × 100k symbols). Indexed-lookup p95 within SOLO-13 §3.1 budgets. GC sweep keeps the steady state bounded. | Schema in SOLO-08 §3 / §4 stands. OQ-S006 marked resolved with the measured numbers. | **Yellow.** Linear growth confirmed but disk/p95 lands within 2× of target at the 50-branch population. | Schema stands; SOLO-13 §3 rows are tightened or relaxed against the measured curve. A note in OQ-S006 records the curve and the M1 re-verification expectation. No schema change. |
| **Red.** Row growth is super-linear, GC sweep is unbounded, or disk/p95 exceeds 2× target. | File an ADR for the delta-override schema (`node_branch_overrides` keyed on branches whose content diverges from a per-repo baseline; sketched as "Option C" in the design review of 2026-05-08). The override schema replaces the relevant SOLO-08 §3.1 / §3.2 sections **before M1's m1.03 begins**. m1.08 still re-verifies the chosen schema on real-repo data. |

## Epic m0.01 — Spike harness

**Tracker:** `engram-v2:m0.01-substrate-spike`

**DoD**
- [ ] Throwaway Go program in `tools/loadtest/spikes/sqlitevec/`
  that:
  - Generates synthetic 768-dim float32 vectors with realistic L2
    norms (e.g. drawn from the empirical distribution of
    `nomic-embed-text` outputs on a corpus snippet).
  - Loads them into a `vec0` virtual table on the reference
    laptop.
  - Runs k=10 and k=50 queries; records p50, p95, p99, max latency
    warm and cold.
  - Computes recall@10 and recall@50 against a 100-pair hold-out
    set.
  - Records peak RSS and on-disk shadow-table size.
- [ ] Output is a single Markdown report committed to
  `tools/loadtest/spikes/sqlitevec/RESULTS.md`.
- [ ] Code is throwaway: no `internal/`, no domain types, no
  production paths. Deleted (or moved to an attic) once M0 closes.

**Acceptance**
- The four exit-gate numbers are produced and signed-off.
- The report cites the sqlite-vec version, the SQLite version, and
  the host platform string.

**Sub-issues**
- `m0.01.1-vector-gen` — synthetic vector generator with realistic
  norms.
- `m0.01.2-load` — load loop into `vec_nodes`; record load time and
  disk growth.
- `m0.01.3-query-bench` — warm/cold query bench at 50k and 1M.
- `m0.01.4-recall-eval` — hold-out generation; recall@k computation.
- `m0.01.5-report` — write `RESULTS.md` with all four numbers and
  the outcome bucket (green/yellow/red).

## Epic m0.02 — Branch-in-PK schema spike

**Tracker:** `engram-v2:m0.02-branch-pk-spike`

**DoD**
- [ ] Throwaway Go program in `tools/loadtest/spikes/branchpk/`
  that:
  - Opens a fresh `engram.db` against the SOLO-08 §3 schema (the
    schema migrations are vendored into the spike from
    `internal/infrastructure/sqlite/migrations/` if available, or
    inlined if M0 runs before m1.03.1 lands — the schema is
    reproduced from SOLO-08 §3.1/§3.2 verbatim in either case).
  - Loads a synthetic population: 100k symbols × 50 branches with
    a configurable per-branch overlap (10%, 30%, 60% dirty).
    Uses direct `INSERT` loops, not the promotion pipeline; the
    measurement is about row-count and storage cost, not about
    parser fidelity.
  - Loads a parallel `findings` population at one finding per 100
    symbols on each branch (per SOLO-04 §8.1's branch-in-PK rule).
  - Runs warm `eng_get_node`-shaped lookups
    (`SELECT * FROM nodes WHERE node_id=? AND branch=?`) and
    `eng_get_edges`-shaped lookups
    (`SELECT * FROM edges WHERE src_node_id=? AND branch=? AND kind=?`)
    against the multi-branch population; records p50/p95/p99.
  - Runs the branch-GC sweep (`DELETE FROM nodes WHERE branch
    NOT IN (...)` plus the matching `edges` and `findings`
    cascades) against a synthetic "10 branches deleted" scenario;
    records wall-clock and disk reclaim.
  - Records on-disk size, WAL size, and process RSS at each
    population.
- [ ] Output is a single Markdown report committed to
  `tools/loadtest/spikes/branchpk/RESULTS.md`.
- [ ] Code is throwaway: no `internal/`, no domain types, no
  production paths. Deleted (or moved to an attic) once M0
  closes.

**Acceptance**
- The four exit-gate numbers for the schema spike are produced
  and signed-off.
- The report cites the SQLite version and the host platform
  string.
- The report carries the synthetic-population assumptions
  (overlap %, finding density) so M1's m1.08 can compare its
  real-repo numbers against the same assumptions.

**Sub-issues**
- `m0.02.1-schema-loader` — synthetic node/edge/finding loader
  with configurable branch overlap.
- `m0.02.2-query-bench` — indexed-lookup p95 across populations.
- `m0.02.3-gc-bench` — branch-GC sweep wall-clock and reclaim.
- `m0.02.4-report` — write `RESULTS.md` with the four numbers and
  the outcome bucket (green/yellow/red).

## Why before M1

Both measurements gate decisions M1 builds on. Taken at the M1
exit gate, a "sqlite-vec doesn't work" or "branch-in-PK doesn't
scale" result triggers a fallback design while M1 code already
sits on top of the failing assumption. Taken at M0, the
fallback ADR is written against the measurement, with no code
to rewrite.

## M0 Close

**Closed:** 2026-05-11

### Spike outcomes

| Spike | Verdict | Commit |
|---|---|---|

### sqlite-vec verdict: RED — CEILING

The vec0 brute-force ceiling measured at **100,000 embedded nodes** on the
reference laptop — well below the 250,000-node threshold required for the
RED-CEILING gate to pass. The latency and recall numbers at 50k were
acceptable, but the ceiling constraint is binding regardless.

Per the §Outcomes table, the HNSW pivot ADR (OQ-S003) is **promoted from M3
work to M1 work** and is mandatory before m1.03 begins.

Amended documents:
- `docs/design/adr/ADR-S0001-sqlite-substrate.md` — status: amended; measured ceiling (100k nodes) and spike commit `4d63d34` cited.
- `docs/design/adr/ADR-S0014-hnsw-pivot.md` — NEW; status: accepted; mandates HNSW backing before m1.03 begins.
- `docs/design/15-glossary/open-questions.md` — OQ-S001 resolved; OQ-S003 pivot trigger pinned at 100k nodes; OQ-S006 resolved.
- `docs/design/08-data-and-storage/README.md` — OQ-S001 and OQ-S006 resolution noted.
- `docs/design/13-nfr/README.md` — BUDGET(unmeasured) rows replaced with measured M0 numbers.

### branch-in-PK verdict: GREEN

Row growth scales linearly. Measured at 28 branches × 100k symbols:
disk = 1.68 GiB, node lookup p95 = 0.039ms, edge lookup p95 = 0.047ms,
GC wall-clock = 518s reclaiming ~700 MiB. All within SOLO-13 §3.1 budgets.

Schema in SOLO-08 §3 / §4 stands. OQ-S006 marked resolved with measured
numbers. m1.08 re-verifies on real-repo data at M1 close.

### M1 readiness statement

**M1 may proceed.** The branch-in-PK schema is confirmed and no schema
amendments are needed. However, the sqlite-vec RED-CEILING verdict is a hard
gate: the HNSW pivot epic (`m1.hnsw-pivot`, implementing ADR-S0014) **must
complete before m1.03 begins**. The M1 epic dependency graph has been amended
to reflect this prerequisite. All other M1 epics are unaffected and may
proceed in their existing dependency order.
