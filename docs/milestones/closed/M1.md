# M1 - Substrate Foundation

**Goal:** A layered V2 codebase with SQLite + sqlite-vec storage,
the save/promote split, an fsnotify watcher, tree-sitter parsers for
Go and TypeScript, ten MCP tools, `veska doctor`, and an
append-only audit log. Every performance budget is measured on
the reference laptop.

**Dependencies:** M0 closed on 2026-05-11 with a **RED-CEILING** verdict on
the sqlite-vec substrate (vec0 ceiling = 100,000 nodes; required ≥ 250,000;
spike commit `4d63d34`) and a **GREEN** verdict on the branch-in-PK schema
(spike commit `72d6ca4`). Per M0 §Outcomes, the RED-CEILING verdict promotes
the HNSW pivot from M3 to M1: **`m1.hnsw-pivot` (ADR-S0014) must complete
before m1.03 begins**. The branch-in-PK schema stands; all other M1 epics
proceed in their existing dependency order. Updated ADRs and docs: ADR-S0001
(amended), ADR-S0014 (accepted), OQ-S001 (resolved), OQ-S003 (pivot trigger
pinned at 100k nodes), OQ-S006 (resolved), SOLO-08, SOLO-13.

## Exit gates (measured)

- [x] Cold-scan 100k LOC repo on the reference laptop in < 60s. → **1.616s PASS**
- [x] `find_symbol` warm p95 < 50ms. → **0.072ms PASS**
- [x] Post-commit hook return p95 < 100ms. → **0.116ms PASS**
- [x] Daemon RSS at steady state ≤ 2 GiB. → **17 MiB PASS**
- [x] Promotion of a 50k-symbol refactor commits in < 5s p95. → **3.83s PASS**
- [x] `semantic_search` p95 < 100ms at 50k vectors against the integrated path. → **1.90ms PASS** (OQ-S001 resolved)
- [x] All tests pass with `-race`. → **36 packages PASS**
- [x] `golangci-lint` clean; `layercheck` clean. → **0 issues PASS**

If a gate misses, the milestone is not done. We rewrite the budget
and re-run, or we record the miss as an ADR.

## Reference hardware

8-core, 16 GiB RAM, NVMe SSD. M1 MacBook Air or equivalent x86-64
laptop. Numbers measured on this tier are normative.

## Epic m1.hnsw-pivot - HNSW vector index pivot

**Tracker:** `solov2-d4v.7` (closed)

**Mandatory gate:** This epic must be closed before m1.03 begins. See ADR-S0014
and M0 §M0 Close for the RED-CEILING verdict that mandates this work.

**Context:** M0 measured the sqlite-vec (vec0) ceiling at 100,000 embedded nodes
on the reference laptop - below the 250,000-node threshold. ADR-S0014 (accepted)
mandates switching to an HNSW-backed vector index that preserves the single-tarball
backup story and the embedded-process deployment model.

**DoD**
- [x] HNSW backing store selected per ADR-S0014 evaluation criteria (candidates:
  `usearch` via cgo, `coder/hnsw` pure Go, or `lancedb` with Go bindings;
  decision recorded in ADR-S0014 §Implementation). → usearch/float16
- [x] `VectorStorage` port implemented against the chosen HNSW backing.
- [x] HNSW index persists to `~/.veska/` alongside `veska.db`; included in
  `veska backup create` tarball.
- [x] Recall@10 ≥ 0.95 at 50k vectors; recall@10 ≥ 0.85 at 250k vectors on
  the reference laptop. → 0.987 @50k, 0.954 @250k
- [x] `semantic_search` p95 ≤ 100ms at 50k vectors and ≤ 150ms at 250k vectors
  on the reference laptop (measured). → 1.90ms @50k, 4.28ms @250k
- [x] `veska doctor storage` reports HNSW index path, size, and vector count.
- [x] ADR-S0014 §Implementation section completed with the chosen backing,
  measured recall, and latency numbers; spike commit hash cited.

**Candidate evaluation matrix for m1.hnsw-pivot.1**

Each candidate must be scored on all five dimensions before a choice is made:

| Dimension | Target | Notes |
|---|---|---|
| Recall@10 at 50k | ≥ 0.95 | hard floor |
| Recall@10 at 250k | ≥ 0.85 | hard floor |
| p95 latency at 250k (k=10) | ≤ 100ms | hard floor |
| **Backup round-trip** | pass | `Save` → tar → `Load` must reproduce identical query results; test with 5 hold-out queries before and after round-trip |
| **Index file size at 250k** | ≤ 1 GiB float32; note float16/int8 option | usearch supports quantization - measure float32, float16, int8 and record recall delta |
| CGo required | note | pure Go preferred if recall/latency equivalent |
| Index build time at 250k | note | not a gate, but recorded |

**Acceptance**
- Measured recall@10 and p95 latency at 50k and 250k vectors recorded in
  ADR-S0014 §Implementation.
- `veska backup create` tarball round-trips through `veska backup verify`
  with the HNSW index present.
- `make build test` green; `golangci-lint` clean.

**Sub-issues**
- `m1.hnsw-pivot.1-backing-selection` - evaluate usearch, coder/hnsw, and
  lancedb against the matrix above; record decision in ADR-S0014; prototype
  both top candidates if recall/latency is within noise.
- `m1.hnsw-pivot.2-port-impl` - implement `VectorStorage` against chosen backing.
- `m1.hnsw-pivot.3-persistence` - HNSW index persistence and backup integration.
- `m1.hnsw-pivot.4-bench` - measure recall@10 and p95 at 50k and 250k; record
  numbers in ADR-S0014.
- `m1.hnsw-pivot.5-doctor` - `veska doctor storage` HNSW reporting.

**Sub-issues**
- `m1.04.1-staging` - `StagingArea` with overlay reads.

  `bootstrap/daemon.go`: `readDB` (unlimited), `writeDB.hot`
  (`MaxOpenConns=1`), `writeDB.embed` (`MaxOpenConns=1`). PRAGMA
  setup helper applies `journal_mode=WAL`, `synchronous=NORMAL`,
  `busy_timeout` per role (SOLO-11 §10, ADR-S0011). Repository
  adapters open transactions on `writeDB.hot`; embed worker
  uses `writeDB.embed`. Caller deadlines surface as
  `ErrBusy` with `cause="seal_in_flight"` or `cause="pool_wait"`
  per SOLO-09 §4.6. Test: promotion in flight +
  concurrent `eng_close_finding` returns the documented error
  classes; embed batch holding lock does not exceed
  `writeDB.hot` `busy_timeout` for short MCP writes.
- `m1.04.8-startup-resync` - on daemon start, after migrations
  and before listener accepts: for each `repos` row, compare
  `last_promoted_sha` with `git rev-parse HEAD`; if HEAD ≠ sha
  and sha is reachable from HEAD, replay
  `git log <sha>..HEAD --reverse` through the promotion pipeline.
  Divergent SHAs (force-push, history rewrite) emit
  `ErrPromotionDivergent` and trigger a fresh full reparse, then
  set `last_promoted_sha = HEAD`. Surface state in
  `veska doctor pipelines.resync` (SOLO-13 §2.1.6a).
  Reads return with `degraded_reasons:["startup_resync"]`;
  writes return `ErrDaemonStarting` until done. Test:
  daemon killed mid-promotion → restart replays the missed commit;
  force-push branch → divergent path takes the fresh-reparse
  branch and clears. (SOLO-03 §5.7, SOLO-08 §5.1.)
