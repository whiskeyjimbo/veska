# M5 — Optional Review Pipeline

**Goal:** An optional LLM-driven review pipeline (security,
contract drift) runs as a goroutine after promotion. Off by default.
Honest cost story in the docs. Findings surface via MCP; the
human-action gate applies.

**Dependencies:** M3.

## Exit gates

- [x] Review pipeline disabled by default. Enabling requires explicit `review.enabled = true`. → `solov2-nz2.2` (config default `review.enabled=false`; no review row enqueued when disabled).
- [x] V2.0 ships **one** `LLMGenerator` impl: `ollama`. Setting `provider` to anything else fails fast at startup. Hosted-provider impls (`anthropic`, `openai`, `gemini`, `openai_compatible`) and the env-var-key resolution they require move to V2.0.1. → `solov2-nz2.1` (startup gate refuses non-`ollama` provider).
- [ ] `veska doctor egress` reports the local Ollama endpoint whenever review is enabled. — **pending: `solov2-jcq`** (not in the solov2-nz2 epic scope).
- [x] Per-promotion token-budget cap enforced; refusal carries `degraded_reasons: ['review_quota_exceeded']`. Tested with a synthetic over-budget promotion. → `solov2-nz2.5`.
- [ ] Documented per-commit *time* budget for one review of a 100-file commit on the reference laptop (m5.00.1). — harness landed by `solov2-zrf`; **reference-laptop number pending a human run** (see Close report).

## Epics

### Epic m5.00 — Local `LLMGenerator` impl (Ollama)

**Tracker:** `veska:m5.00-local-llm`

**DoD**
- [ ] HTTP-based `LLMGenerator` impl in `internal/infrastructure/llm/ollama/`. Posts to a local Ollama endpoint configured via `[llm_generator].endpoint` and `[llm_generator].model` (CONFIG-SURFACE.md).
- [ ] Composition root selects this impl unconditionally at V2.0; setting `provider` to anything other than `"ollama"` fails fast at startup with a clear error pointing at the V2.0.1 deferral.
- [ ] Each call populates `Generation.Provenance` with `(model_id, prompt_template_version, input_hash)` per SOLO-05 §2.5 so cached outputs can be invalidated when the model changes.
- [ ] Per-impl timeout (`llm_generator.timeout` DEFAULT) and 3-retry policy on transient failures (5xx, network).
- [ ] Documented time-budget example for one review of a 100-file commit on the reference laptop. Lives in M5 close report.

**Acceptance**
- Test: starting the daemon with `provider = "anthropic"` (or any non-`ollama` value) exits non-zero with a message naming the V2.0.1 deferral.
- Test: a synthetic review against the local Ollama returns a populated `Generation` with a non-empty `Provenance`.
- Test: a transient 5xx from Ollama triggers exactly 3 retries with exponential backoff before the post_promotion_queue row is marked failed.

**Sub-issues**
- `m5.00.1-ollama-impl` — Ollama HTTP adapter + provenance.
- `m5.00.2-startup-refusal` — fail-fast on non-`ollama` provider with V2.0.1-deferral message.
- `m5.00.3-cost-doc` — time-budget example for a 100-file commit on the reference laptop, included in the M5 close report.

**Deferred to V2.0.1** (filed as separate beads issues, not blocked by M5 close):
- Hosted `LLMGenerator` impls: Anthropic Messages, OpenAI Chat Completions, Gemini generateContent, OpenAI-compatible endpoint adapter.
- Env-var-only API key resolution (`VESKA_LLM_API_KEY` / `ANTHROPIC_API_KEY` / etc.).
- USD cost machinery (per-call cost from provider response, `max_usd_per_day` / `max_usd_per_week` enforcement, `veska doctor egress` redaction tests for hosted endpoints).

### Epic m5.01 — Review goroutine

**Tracker:** `veska:m5.01-review-goroutine`

**DoD**
- [ ] post-promotion queue `work_kind='review'`; one goroutine drains it.
- [ ] `LLMGenerator` interface dispatches to local Ollama or configured remote endpoint.
- [ ] Failure retries up to 3.
- [ ] On persistent failure, the row stays `failed` (no auto-ack)
  and a `Finding` is emitted with
  `rule='review-pipeline-failure'`, `severity='high'`,
  `source_layer='quality'`, anchored to the promotion's commit.
- [ ] Closing that finding (human-action-gated) flips the post-promotion queue row to
  `done`. ADR-S0004 owns the policy.
- [ ] `veska doctor post-promotion-queue` exits 2 if any `review` row is
  `failed` without an open companion finding.

**Sub-issues**
- `m5.01.1-lane` — review post-promotion queue lane.
- `m5.01.2-dispatch` — `LLMGenerator` invocation; timeout config.
- `m5.01.3-retry` — retry policy (3 attempts, exponential backoff).
- `m5.01.4-failure-finding` — emit `review-pipeline-failure` finding; wire close-flips-row.
- `m5.01.5-doctor-invariant` — `veska doctor post-promotion-queue` exit-2 check.

## Close report

### Exit gate 5 — per-commit review time budget (m5.00.3 / `solov2-zrf`)

**Measurement procedure.** The repeatable harness lives at
`tools/loadtest/reviewtiming/` and is run via:

```bash
make eval-review-timing
```

It builds a synthetic ~100-file Go commit fixture, drives each file through the
real `internal/application/review` `Handler` (the `WorkKindReview` lane) against
a real local Ollama generator, and reports the total wall-clock budget for the
100-file pass plus the per-file mean. Override `REVIEW_TIMING_FILE_N`,
`VESKA_OLLAMA_URL`, and `VESKA_REVIEW_MODEL` as needed; the harness skips
cleanly when Ollama is unreachable. See `tools/loadtest/reviewtiming/README.md`.

harness). Record the `wall_clock_budget` and `per_file_mean_ms` from the
`REVIEW_TIMING` JSON output here once a reference-laptop run has been done.
