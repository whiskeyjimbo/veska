# M3 — Pipelines and Embedder

**Goal:** Promotion pipeline runs structural checks synchronously; an
async embedder worker keeps `node_embedding_refs` drained;
`semantic_search` runs against the production vector path (sqlite-vec
default, usearch/float16 above the M2-ratified threshold per
ADR-S0015); auto-link suggestions land;
revalidation sweeps invalidate stale findings on content drift.

**Dependencies:** M1, M2 (incl. m2.06 dual-backend ratification).

## Exit gates

- [ ] Embedder throughput on local CPU Ollama: measured at 5–20 emb/s on the reference laptop.
- [ ] `semantic_search` recall measured at 50k and 250k against the production usearch/float16 backend on a fixed eval set; results recorded in the milestone close report. (The HNSW pivot itself was ratified in M2; ADR-S0014's M1 bench already recorded 0.987@50k / 0.954@250k — M3 confirms the number end-to-end on the live `semantic_search` path.)
- [ ] Auto-link false-positive rate measured on a fixture and surfaced in the milestone close report.
- [ ] Revalidation completes in < 60s for a 10k-edge commit.
- [ ] All four post-promotion queue `work_kind` lanes drain to `state='done'` under a fuzz workload.

### Epic m3.04 — Auto-link

**Tracker:** TBD (beads issue created in Stage 2)

**DoD**
- [ ] post-promotion queue `work_kind='auto_link'` proposes `Edge` rows with `confidence='unresolved'` from embedding similarity AND emits one finding per proposed edge (`source_layer='semantic'`) whose anchor is that specific edge. The edge is never inserted silently — it only becomes visible via the finding.
- [ ] Threshold configurable; default tuned on the eval set.
- [ ] No auto-merge; the developer accepts or suppresses.
- [ ] Accept-flow uses existing `eng_close_finding` with
  `reason="accept"`; the close handler promotes the anchor's
  unresolved edge to `confidence='definite'`. Suppress/close paths
  leave the edge at `confidence='unresolved'` (visible only to
  queries that opt into unresolved edges). No new MCP tool.

**Sub-issues**
- `m3.04.1-similarity` — top-k similarity per node.
- `m3.04.2-finding-shape` — auto-link finding kind.
- `m3.04.3-accept-flow` — `eng_close_finding[reason="accept"]` promotes the linked edge.
- `m3.04.4-fp-measurement` — false-positive rate on fixture.

### Epic m3.05 — Revalidation

**Tracker:** TBD (beads issue created in Stage 2)

**DoD**
- [ ] post-promotion queue `work_kind='revalidate'` re-checks open findings whose anchor `content_hash` changed.
- [ ] Per-rule dispatch on stale findings:
  - `dead-code`, `contract-drift`: cheap re-run on current node state. If the rule still fires, REFRESH `anchor_content_hash` in place (state stays `open`, no closed_reason). If the rule no longer fires, transition to `closed` with `closed_reason='revalidated_obsolete'`.
  - `auto-link`: always close as `revalidated_obsolete` (similarity re-run is heavy and out of scope for m3.05).
  - any other rule: conservative close as `revalidated_obsolete`.
- [ ] No `closed_reason='superseded_by_revalidation'` is ever written. `finding_id` = `hash(rule, anchor)` is branch-stable, so a re-firing rule produces the SAME row — refreshing the anchor hash is the supersede path. (SOLO-04 §8.1.)
- [ ] Sweep runs after promotion; bounded to the symbols touched by the promotion.

**Sub-issues**
- `m3.05.1-anchor-tracking` — finding ↔ node hash linkage.
- `m3.05.2-sweep` — revalidate goroutine.
- `m3.05.3-supersede` — chain old → new finding.
- `m3.05.4-bench` — 10k-edge commit revalidation timing.
