# Research: Ideas from Chizu competitive analysis (May 2026)

Source: https://github.com/vectorian-rs/chizu (v0.3.0, last checked 2026-05-13)

Each idea below is a stub. Promote to a feature doc under `docs/features/active/` when
it's time to design it properly. Notes on v2 roadmap overlap included.

---

## 1. Task-intent classification before retrieval

**What it is:** Classify the incoming natural-language query into one of N task categories
(understand, debug, build, test, deploy, configure, implement, fix, bench, release, setup,
architecture) before hitting the vector index, then boost entity-kind weights per category.
E.g. "how do I deploy?" up-weights Dockerfiles and Terraform roots; "where is X tested?"
up-weights test files.

**Chizu impl:** 12 hard-coded categories, per-category `HashMap<EntityKind, priority>`.
Classifier is a lightweight embedding cosine match against category exemplars.

**v2 overlap:** M3 `semantic_search` / M4 MCP expansion. The `semantic_search` tool is
the natural insertion point. No intent layer exists yet.

**Sketch:**
- Add a `query_intent` classifier step to `semantic_search` (or expose as a separate
  `classify_query` MCP tool)
- Maintain a config-level or hard-coded priority table per intent class
- Boosts compose with existing score before ranking; don't replace

## 2. Infrastructure and doc entities as first-class graph nodes

**What it is:** Model Dockerfiles, Terraform roots, docker-compose files, CI/CD workflows,
SQL migrations, README/spec docs, and agent config files (CLAUDE.md, AGENTS.md) as typed
`Node` variants with proper edges (`Deploys`, `Migrates`, `ConfiguredBy`, `Builds`, `Docs`).

**Chizu impl:** 18 edge kinds total; infra/doc kinds are `InfraNode`, `DocNode`,
`AgentConfig`, `CiConfig`, `Migration`. Scanner-based (not AST) for most.

**v2 overlap:** M1 domain model â€” `Node` and `EdgeKind` are defined in
`internal/core/domain/`. Adding new `NodeKind` variants is a domain-layer change.
Could land as a separate M1.x epic or be deferred to M2/M3.

**Sketch:**
- New `NodeKind` enum variants: `DocFile`, `InfraFile`, `CIConfig`, `AgentConfig`
- New `EdgeKind` variants: `Deploys`, `Migrates`, `ConfiguredBy` (on top of current 5)
- Lightweight file-pattern scanner (no AST needed): `Dockerfile*`, `*.tf`, `.github/workflows/*.yml`, `CLAUDE.md`
- Index these in the same promotion pipeline; embeddings optional

## 3. LLM-generated entity summaries

**What it is:** Each node gets a short prose summary generated by a local LLM (e.g.
`llama3:8b`), stored in the `nodes` table. Feeds both the reranker and the reading-plan
output so agents get interpretable results without opening files.

**Chizu impl:** Summary stored as a column in SQLite; generated at index time via Ollama;
refreshed when content hash changes.

**v2 overlap:** M5 `LLMGenerator` port (`core/ports/`) is exactly the abstraction needed.
The `review goroutine` in M5 already uses Ollama; summary generation is a parallel use case
that could share the same `LLMGenerator` interface with a different prompt set.

**Sketch:**
- Add `summary TEXT` column to `nodes` schema (nullable; filled async)
- `work_kind='summarize'` in `post_promotion_queue`
- `LLMGenerator.Summarize(node Node) (string, error)` method
- Return summaries in `find_symbol`, `get_node`, `semantic_search` results

## 4. Interactive HTML graph visualization

**What it is:** `veska visualize` generates a self-contained HTML file with an
interactive explorer: search, breadcrumbs, node inspector pane, theme toggle, and
`vscode://file/...` deep links to jump directly to symbols in the editor.

**Chizu impl:** Static asset bundle embedded in the binary (JS + CSS); data serialized
as JSON inline in the HTML. Also supports a simpler SVG-only mode.

**v2 overlap:** M4 wiki area â€” wiki already renders to `docs/veska/`. Visualization
could be an additional `veska wiki --format=html` output, or a standalone
`veska visualize` command. Pure read path; just needs a `GraphStorage.Query` call.

**Sketch:**
- `veska visualize [--output docs/veska/graph.html]`
- Serialize nodes + edges to embedded JSON; inline a minimal graph renderer (d3 or
  vanilla canvas)
- Include `vscode://file/<abs_path>:<line>` links on each node
- Make it a `work_kind='visualize'` in the wiki pipeline so it regenerates on promotion

## 5. Score-gap cutoff on search results

**What it is:** After retrieval, trim the tail of results where the relevance score drops
sharply (score gap > threshold between consecutive results). Prevents low-confidence noise
from reaching the agent.

**Chizu impl:** Configurable `score_gap_cutoff` float in `.chizu.toml`; applied after
reranking, before returning results.

**v2 overlap:** M3 `semantic_search` â€” currently returns a fixed top-k from the vector
index with no quality floor. This is a small, self-contained improvement.

**Sketch:**
- Add `score_gap_cutoff float64` to `SearchOptions` / `config.toml` (`[search]` section)
- After sorting results by score, iterate pairwise; drop everything after first gap >
  threshold
- Default: disabled (0.0) so behavior is unchanged until opted in
- Expose in `semantic_search` MCP tool as optional `score_gap_cutoff` param

## 6. Configurable retrieval weights

**What it is:** Expose per-signal weights (vector, keyword/BM25, name-match, path-match,
export-bias) in `veska.toml` so repos can tune search quality without code changes.
Weights sum to 1.0; validated at startup.

**Chizu impl:** `[search.rerank_weights]` TOML table with 6 floats; validated at load.

**v2 overlap:** M2 config surface (`operations/CONFIG-SURFACE.md`) and M3
`semantic_search`. The config infrastructure will exist by M2; the weights themselves
land in M3 when the multi-signal search path is built.

**Sketch:**
- Add `[search]` section to `veska.toml` schema:
  ```toml
  [search]
  weight_vector   = 0.6
  weight_keyword  = 0.25
  weight_name     = 0.10
  weight_path     = 0.05
  score_gap_cutoff = 0.0
  ```
- `SearchOptions` struct carries the weights; defaults hardcoded; config overrides
